## Incident 2026-07-17 — sleep script caught in instant-retrigger loop, paused autonomous loop
- Called the sleep script twice in a row; both times it returned in under a second reporting the same file (`outbox/question-project-switch-visual-behavior.md`, still unanswered) as "waiting." Read `scripts/await-messages-random.js` source to confirm root cause rather than guessing: `checkDirectories()` runs once synchronously before the real wait loop starts, and treats *any* `.md`/`.json` file present in a watched dir as "new" — it has no memory of what it already reported last time it ran (each invocation is a fresh process, no state file).
- This is a real bug, not a fluke: since Step 4 requires always keeping a pending question in `outbox/` until answered, every subsequent sleep-script call would instantly exit and report the same unanswered file, forever — a genuine runaway-loop risk (each "instant wake" is a full new agent turn, not a cheap no-op). Filed **`[BUG]-overseer-poller-instant-retrigger-on-pending-outbox.md`** with root cause and 3 concrete fix options (state-file signature tracking is the most robust; content-based Answer-field check is Overseer-specific and simpler).
- **Decision: paused the autonomous sleep/wake loop rather than keep calling the broken script.** No tool available at the time gave a genuine long real-world delay that fit the situation. Continuing to mechanically re-call the broken poller would have directly violated the "don't burn API tokens" instruction in spirit.
- **Resume plan**: once the human either (a) answers the pending outbox question, (b) fixes the poller bug via the filed ticket, or (c) explicitly tells the Overseer to resume, the loop can restart normally.

## Pattern note: writing a question then immediately sleeping always self-triggers once
- This is expected behavior of the fixed poller, not a regression: the state file only records a signature the first time it *sees* a file, so a freshly-written outbox question (never seen before) will always cause the very next sleep-script call to report it as "waiting" — even though it's unanswered and just self-written. If the file just written has an empty `Answer:`, there's nothing to process — immediately re-call the sleep script and it does a genuine wait the second time (signature is now recorded). This same pattern recurred later (2026-07-17, QA-stall-question cycle) and was recognized immediately from this precedent.
