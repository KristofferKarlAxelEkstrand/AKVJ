## Sweep 2026-07-17 (random wake, ~239s later) — Task 80 closed the loop, found a real bug I missed
- Task 80 (MIDI clock audit) completed and QA-approved: "bug found + fixed... stale `#lastTime` bug." Investigated to close my own watch-list item and understand what I missed.
- The bug: `Clip.js#advanceFrame` — when a BPM-synced clip is being driven by clock pulses (`bpmSource === BPM_SOURCE_CLOCK`), time-based advancement is skipped, but `#lastTime` (the clip's own last-rendered timestamp for *time*-based advancement) wasn't being cleared. If the MIDI clock ever stopped and `bpmSource` fell back to CC/default (e.g. clock timeout, per `AppState`'s `#resetClockTimeout`), the clip would resume time-based advancement using a *stale* `#lastTime` from potentially long ago (last time it was in time-based mode, could be app startup) — computing a huge `elapsed` value and jumping/skipping through many frames instantly. Fixed by setting `this.#lastTime = null` every time the clock-sync branch is taken (`Clip.js:150-153`), so a fallback to time-based mode always starts fresh instead of computing a bogus elapsed time. Verified this fix is in place and looks correct.
- **Calibration lesson**: my own earlier MIDI-clock review (a few cycles ago) checked `AppState.js`'s internal clock/BPM state consistency (Start/Stop/Continue, rolling average) but didn't trace the *downstream ripple effect* on `Clip.js` when `bpmSource` transitions between clock and non-clock — a cross-file interaction I missed. Worth remembering for future robustness reviews: check not just "is this module's own state consistent" but "what happens to every consumer when this module's externally-observable state changes out from under it."
