# AGENTS.md Writing Guide

Detailed guide for writing effective AGENTS.md files. Based on the [agents.md](https://agents.md/) open standard and community best practices.

## What AGENTS.md Is

AGENTS.md is a markdown file at the root of your repository that provides AI coding agents with persistent, project-specific operational guidance. Think of it as onboarding documentation for AI — the same information you'd give a new team member on their first day.

It is **not**:
- A replacement for `README.md` (which is for humans)
- A place for personal preferences (use `CLAUDE.md` or equivalent)
- A place for task-specific workflows (use `SKILL.md` skills)
- Long architectural documentation (keep it concise)

## Recommended Sections

### 1. Project Overview

What the project is, primary language, framework, and versions. Keep it to 2-3 sentences.

```markdown
## Project Overview

AKVJ is a real-time VJ (Video Jockey) application for live performance visuals.
It uses vanilla JavaScript (ES6+ modules), HTML5 Canvas (240x135 at 60fps),
Web MIDI API (Chrome/Chromium only), Custom Elements, and Vite.
```

### 2. Quick-Start

Steps an agent should take before starting work. Numbered, actionable.

```markdown
## Agent Quick-Start

1. **Read this entire file** before starting work
2. **Install dependencies**: `npm install` (takes ~13 seconds, never cancel)
3. **Make minimal changes**: Focus on surgical, precise modifications
4. **Validate frequently**: Run `npm run lint && npm run test && npm run build`
```

### 3. Critical Constraints

Hard rules the agent must follow. Performance requirements, architecture rules, things agents must not do.

```markdown
## Critical Constraints

- **60fps rendering** — Never introduce blocking operations in the render loop
- **Vanilla JS only** — Do not introduce frameworks
- **Chrome/Chromium only** — Web MIDI API dependency
```

### 4. Build and Test Commands

Exact commands with flags. Not vague tool names. Include environment setup and dev server startup.

```markdown
## Common Commands

npm run dev              # Start development server (localhost:5173)
npm run build            # Production build
npm run test             # Run unit tests with Vitest
npm run test:visual      # Run visual regression tests (Chromium)
npm run lint             # Check code quality
```

### 5. Code Conventions

Only rules that differ from language defaults. Things the agent would get wrong without guidance.

```markdown
## Code Conventions

- **Private fields**: `#fieldName` for all internal state
- **Timing**: Use `performance.now()`, not `Date.now()`
- **Dev-only logging**: Wrap `console.log` in `import.meta.env.DEV` check
```

### 6. Testing Instructions

How to run the suite, a single test, and what to mock.

```markdown
## Test Structure

- Unit tests: `npm run test` (jsdom environment)
- Visual tests: `npm run test:visual` (real Chromium browser)
- Single test: `npx vitest run -t "test name pattern"`
- Mock MIDI using `test/utils/fake-midi.js`, never expect real hardware
```

### 7. Security/Boundaries

What the agent should never touch. Files, directories, or patterns that are off-limits.

```markdown
## Security Considerations

- Never modify files in `src/public/animations/` (generated by build pipeline)
- Never commit `.env` files
- Path traversal protection in AnimationLoader must not be weakened
```

### 8. Commit and PR Guidelines

Branch naming, commit format, merge strategy.

```markdown
## Commit Guidelines

- Conventional commits: `feat:`, `fix:`, `chore:`, `docs:`
- Squash merge only
- Branch naming: `feature/description`, `fix/description`
```

## Size Guidelines

- Keep under **32 KiB** (Codex default limit)
- Keep under **150 lines** recommended (community consensus)
- For monorepos, use sub-directory `AGENTS.md` files instead of one giant file
- Move detailed reference material to `references/` files or skills

## Hierarchy

Both Claude Code and Codex support hierarchical AGENTS.md:

```
repo/
├── AGENTS.md              # Root instructions (always loaded)
├── src/
│   ├── AGENTS.md          # Source-specific instructions
│   └── package/
│       └── AGENTS.md      # Package-specific instructions
```

The **nearest file wins** — each subproject ships tailored instructions.

## What Not to Put in AGENTS.md

- Personal response style preferences (use `CLAUDE.md` or tool-specific config)
- Task-specific workflows like "how to deploy" (use `SKILL.md` skills)
- Long architectural descriptions (use `references/` files)
- Philosophical guidance like "write clean code" (be specific instead)
- Information already in `README.md` (reference it instead of duplicating)

## Tool Compatibility

AGENTS.md is read natively by 30+ agents including:

- OpenAI Codex (primary format)
- Claude Code (as fallback when `CLAUDE.md` is absent, or via `@AGENTS.md` import)
- GitHub Copilot
- Cursor
- Gemini CLI (via config)
- Windsurf
- Aider (via config)
- Devin
- Zed
- Warp
- And many more

For Claude Code specifically, create a `CLAUDE.md` with `@AGENTS.md` as the first line to ensure it reads your AGENTS.md content.
